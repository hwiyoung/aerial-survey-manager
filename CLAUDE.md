# 개발 플로우 규칙

## 작업 진행 방식
- 모든 작업은 반드시 `docs/PROGRESS.md` 파일을 참조하고 업데이트하며 진행한다.
- 각 단계를 완료할 때마다 PROGRESS.md의 해당 체크박스를 체크하고, 다음 단계로 자동 진행한다.
- **게이트 단계**(🚪 표시)에서는 반드시 멈추고 사용자 확인을 요청한다.
- 검토 단계에서 문제를 발견하면: 문제 내용을 PROGRESS.md에 기록하고, 해당 단계를 반복한다. 이전 단계로 돌아가야 할 경우 PROGRESS.md에 "🔄 [N단계]로 회귀: 사유" 형식으로 기록한다.

## 단계 정의

### Phase A: 계획 (사용자 확인 필요)
- A1. 요구사항 분석 및 구현 계획 수립
- A2. 계획의 논리적 타당성 자체 검토  
- A3. 계획이 과도하지 않은지 (오버엔지니어링) 검토
- 🚪 **GATE: 계획을 사용자에게 보고하고 승인받기**

### Phase B: 구현 (자동 진행)
- B1. 계획에 따라 구현
- B2. 구현이 원래 목적에 부합하는지 검토
- B3. 잠재적 버그, 크리티컬 이슈, 보안 문제 검토 및 수정
- B4. 수정 사항에 새로운 문제가 없는지 검토

### Phase C: 코드 품질 (자동 진행)
- C1. 과도하게 큰 함수/파일 분리
- C2. 기존 코드와 통합/재사용 가능한 부분 검토
- C3. 사이드 이펙트 확인
- C4. 불필요한 코드 정리
- C5. 코드 품질 최종 검토
- 🚪 **GATE: 변경 사항 요약을 사용자에게 보고**

### Phase D: 최종 검증 (사용자 확인 필요)
- D1. 사용자 흐름(UX) 관점 검토
- D2. 전체 변경 사항 통합 검토
- D3. 배포 가능 수준인지 최종 판단
- 🚪 **GATE: 최종 보고 후 커밋/PR 승인 요청**
- D4. 커밋 및 PR 작성

## 보고 형식
각 게이트에서 보고할 때는 다음 형식을 사용한다:
```
📍 현재 단계: [단계명]
✅ 완료 항목: [목록]
⚠️ 발견된 이슈: [있다면]
📋 다음 단계: [예정 작업]
➡️ 필요한 결정: [사용자에게 물어볼 것]
```
