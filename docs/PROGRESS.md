```
## Step 3: 실제 작업 시작하는 방법

작업을 시작할 때, 에이전트에게 보내는 **프롬프트는 딱 하나**입니다:
```
[기능 설명 또는 스케치/계획안 첨부]

위 내용을 구현해줘.
CLAUDE.md의 개발 플로우를 따라 진행하고, 
docs/PROGRESS.md에 새 작업 섹션을 만들어서 진행 상황을 추적해.
첫 번째 게이트까지 자동으로 진행한 다음 나에게 보고해.
```

**이게 전부입니다.** 이 한 번의 프롬프트로 에이전트가 A1→A2→A3까지 스스로 진행하고, 게이트에서 멈춰서 보고합니다.

## Step 4: 게이트에서 사용자가 할 일

에이전트가 게이트에서 멈추면, 당신이 해야 할 일은 세 가지 중 하나입니다:

**상황 1: 계획이 좋다** →
```
승인. 다음 게이트까지 진행해.
```

**상황 2: 일부 수정 필요** →
```
계획에서 [X 부분]을 [Y]로 변경해. 
수정 후 다음 게이트까지 진행해.
```

**상황 3: 계획이 근본적으로 잘못됨** →
```
계획을 재수립해야 해. 이유: [사유]. 
A1부터 다시 시작하고, PROGRESS.md에 회귀 사유 기록해.
다시 첫 번째 게이트까지 진행해.
```

> 핵심: "돌아가야 하나 말아야 하나"를 고민할 필요 없이, **항상 "다음 게이트까지 진행해"로 끝내면 됩니다.** 에이전트가 중간 단계는 알아서 합니다.

---

## Step 5: 전체 흐름 요약 (당신이 하는 것 vs 에이전트가 하는 것)

| 당신이 하는 것 | 에이전트가 하는 것 |
|---|---|
| 요구사항/스케치 전달 + "첫 게이트까지 진행해" | A1→A2→A3 자동 진행 후 게이트에서 보고 |
| 계획 검토 후 "승인, 다음 게이트까지" | B1→B2→B3→B4→C1→C2→C3→C4→C5 자동 진행 후 보고 |
| 변경사항 확인 후 "승인, 다음 게이트까지" | D1→D2→D3 자동 진행 후 보고 |
| "커밋해" | D4. 커밋 및 PR 작성 |

**결과: 18번 프롬프팅하던 것이 4번으로 줄어듭니다.**

## 추가 팁

**게이트를 더 줄이고 싶다면**, Phase B와 C 사이의 게이트를 제거하고 B~C를 하나로 묶으면 3번만 프롬프팅하면 됩니다. 다만 대규모 변경일 때는 중간 확인이 유용하므로, 작업 규모에 따라 프롬프트에서 조절하세요:
```
# 소규모 작업일 때

# 대규모 작업일 때  
모든 게이트에서 멈춰서 보고해.
